## 개요

> 💡 Phase 1에서 '누가 무엇을 하는지'를 정의했다면, Phase 2는 '어떻게 실행되는지'를 정의한다. `workflow_context.json`이 Cursor AI의 핵심 참조 파일이 된다.

## 2. Agent 계층 구조 (Layer Model)

> 💡 0_3 멀티에이전트 구조 표준 기반. 5개 Layer로 역할을 분리하여 단일 Agent에 기능이 집중되는 것을 방지한다.

### 2.1 Layer 정의 및 역할

| Layer | 명칭 | 역할 | 담당 Agent | Authority |
|-------|------|------|-----------|-----------|
| **L1** | Orchestrator | 전체 Workflow 조정, Escalation 판단, Decision Gate 실행 | ⭐ Orch | L1 / HIGH |
| **L2** | Capability | 실제 기능 수행 — LLM·Tool·Skill 호출, Scope 내 작업 | UIUX-A / Front-A / Back-A | L1 / MED·HIGH |
| **L3** | Quality Control | 테스트 수행, Risk 검증, Guardrail 점검 | Test-A | L1 / MEDIUM |
| **L4** | Monitoring | KPI 모니터링, Drift 탐지, Incident 트리거 | Monitor-A | L0 / LOW |
| **L5** | Evolution | 로그 분석, Context 수정 제안, 재배포 트리거 | Evol-A | L1 / HIGH |

### 2.2 계층 간 통제 규칙

| 규칙 | 내용 | 위반 시 조치 |
|------|------|-------------|
| **Capability → Orchestrator 승인 필수** | L2 Agent는 Orch 승인 없이 직접 실행 권한 불가 | 즉시 실행 차단 + Escalation |
| **Layer 간 직접 권한 상승 금지** | 모든 권한 변경은 Governance Board 경유 필수 | Scope 초과 처리 + 로그 기록 |
| **단일 Agent 기능 집중 금지** | 전략 + 실행 + 모니터링 동시 담당 불가 | 아키텍처 재검토 필수 |
| **Global Context 우선 로드** | 모든 Agent는 실행 전 Global Context 반드시 로드 | 배포 차단 |
| **Evolution → Governance 승인 필수** | L5 Agent는 Context 수정 제안만 가능, 직접 적용 불가 | Context 복원 + 감사 |

## 3. 전체 Workflow Diagram

### 3.1 표준 실행 흐름 (Phase 공통)

아래는 모든 Phase에 공통 적용되는 Agent 실행 흐름이다. Decision Gate(◆)에서 Human 승인이 필요하다.

```
① 요청 입력          ② Context 로드         ③ Orch 수신           ◆ Decision Gate       ④ Capability 실행
Human / 외부 트리거   Global Context +       요청 분석 +           Human 승인            UIUX-A / Front-A /
                     Agent Context          Agent 할당            (P1 Pre-approval)     Back-A

                                                    ▼

⑤ Tool 호출          ⑥ QC 검증              ◆ Post-review          ⑦ Monitoring          ⑧ audit_logs
허용된 Tool만 실행    Test-A Risk 검증       Human 최종             Monitor-A             모든 실행
                                           결과 확정              KPI 기록              결과 기록 필수
```

> ⚠️ **On-risk(P2) 감지 시:** ④~⑥ 즉시 중단 → Orch 에스컬레이션 → Human 조치 → 재개 또는 롤백

### 3.2 Phase별 Workflow 특징

| Phase | 주도 Agent | 핵심 Workflow | 주요 Gate |
|-------|-----------|--------------|-----------|
| **Phase 1 (Vision)** | Orch (보조) | Human 주도 → Agent 보조 → 산출물 검토 | Persona 확정 / R&R 승인 |
| **Phase 2 (Design)** | ⭐ Orch (주도) | Orch가 Workflow 설계 → Human Decision Gate | Workflow 승인 / Gate 설계 승인 |
| **Phase 3 (Build)** | Front-A / Back-A | 코드 생성 → Human 검수 → Orch 통합 | 코드 리뷰 / 배포 승인 |
| **Phase 4 (Test)** | Test-A / Orch | 시나리오 실행 → KPI 측정 → Evaluation Report | Eval Report 확정 / 재배포 승인 |
| **Phase 5 (Deploy)** | Back-A / Monitor-A | Release Gate → 배포 → Monitoring 시작 | Release Gate 통과 |
| **Phase 6 (Evolve)** | Monitor-A / Evol-A / Orch | Drift 감지 → 제안 → Governance 승인 → 재배포 | Context 수정 승인 |

## 4. Decision Gate 정의

> 💡 Decision Gate는 Workflow 진행 중 Human 승인이 필요한 통제 지점이다. Gate를 통과하지 못하면 Agent는 대기 상태를 유지한다.

### 4.1 Gate 목록 및 위치

| Gate ID | Gate명 | 위치 (Phase) | 승인 주체 | HITL 패턴 | SLA |
|---------|--------|-------------|----------|-----------|-----|
| **G-01** | Persona 확정 Gate | Phase 1 (1.2) | UX Designer | P3 Post-review | 48h |
| **G-02** | R&R Matrix 승인 Gate | Phase 1 (1.3) | PM + AI Architect | P3 Post-review | 48h |
| **G-03** | Authority Level 승인 Gate | Phase 1 (1.4) | Business Owner | P1 Pre-approval | 24h |
| **G-04** | Workflow 설계 승인 Gate | Phase 2 (전체) | AI Architect | P1 Pre-approval | 24h |
| **G-05** | 코드 리뷰 Gate | Phase 3 (구현) | Front-H / Back-H | P3 Post-review | 48h |
| **G-06** | DB 변경 승인 Gate | Phase 3 (DB) | AI Architect + DevOps | P1 Pre-approval | 24h |
| **G-07** | Evaluation Report 확정 Gate | Phase 4 (Test) | QA Engineer | P3 Post-review | 48h |
| **G-08** | Release Gate | Phase 5 (배포) | AI Architect + PM | P1 Pre-approval | 24h |
| **G-09** | Context 수정 승인 Gate | Phase 6 (Evolve) | Governance Board | P1 Pre-approval | **24h** |

> ✅ **v1.1 수정:** G-09 SLA `48h` → `24h` (P1 Pre-approval 기준 hitl_context.json 통일)

### 4.2 Gate 통과 / 거부 / 타임아웃 처리

| 결과 | 즉각 처리 | Agent 조치 | audit_logs 기록 |
|------|----------|-----------|----------------|
| ✅ **APPROVED** | Agent 실행 재개 | 다음 단계 진행 | `status=APPROVED, approver, approved_at` |
| ❌ **REJECTED** | Agent 실행 취소 + 요청자 알림 | 거부 사유 분석 → 재설계 또는 폐기 | `status=REJECTED, reason, rejected_by` |
| 🔄 **REVISE** | 수정 사항 명시 → Agent 재작업 | 산출물 수정 → 재검토 요청 | `status=REVISED, revision_note` |
| ⏰ **SLA 초과** | 상위 승인자 자동 에스컬레이션 | Orch 알림 → 리마인더 발송 | `status=ESCALATED, escalated_at` |
| 🚨 **Override** | 즉시 중단 + Snapshot 저장 (3초 내) | 전체 Agent 중단 → Human 조치 대기 | `event=OVERRIDE, triggered_by` |

## 5. Tool Invocation Flow

### 5.1 Tool 목록 및 Agent별 사용 기준

| Tool명 | 사용 Agent | 기능 | 제약 조건 |
|--------|-----------|------|----------|
| **llm_call** | 모든 Capability Agent | LLM API 호출 (분석·생성·추천) | max_tokens 제한 / Prompt Injection 방어 필수 |
| **db_query** | Back-A | PostgreSQL 조회 (SELECT) | 트랜잭션 없이 조회만 / Audit Log 불필요 |
| **db_write** | Back-A | PostgreSQL 쓰기 (INSERT/UPDATE) | 트랜잭션 필수 / Human 승인 (G-06) / Audit Log 필수 |
| **api_call_internal** | Back-A / Orch | 내부 FastAPI 엔드포인트 호출 | RBAC 인증 필수 / useAgent 훅 경유 |
| **api_call_external** | Back-A | 외부 API 호출 | Human 승인 필수 / API Key 환경변수 / Audit Log 필수 |
| **file_read** | 모든 Agent | 파일 시스템 읽기 | 허용된 디렉토리만 / 민감 데이터 마스킹 |
| **file_write** | Back-A / Front-A | 파일 시스템 쓰기 | Human 승인 필수 / Audit Log 필수 |
| **send_notification** | Orch / Monitor-A | 알림 발송 (이메일·슬랙) | 승인 없이 가능 / 수신자 허용 목록 내 |
| **run_test** | Test-A | 테스트 시나리오 자동 실행 | 프로덕션 DB 접근 금지 / 결과 trajectory_logs 기록 |
| **read_logs** | Monitor-A / Evol-A | audit_logs / trajectory_logs 조회 | SELECT 조회만 / 외부 전송 금지 |

### 5.2 Tool 호출 표준 절차

```
① Tool 호출 요청   → ② Scope 검증        → ③ Risk 체크         → ④ Tool 실행
  Agent가              allowed_tools          HIGH 이상 시            허용 범위 내
  Tool 호출 결정       내 여부 확인           Human 승인 요청         실행

                                                        ↓

                                              ⑤ 결과 검증         → ⑥ Audit Log 기록
                                                Output 유효성         필수 — 모든 Tool
                                                확인                  실행 결과 기록
```

> ⚠️ **Scope 외 Tool 호출 시도** → 즉시 차단 → Orchestrator 에스컬레이션 → `audit_logs`에 `SCOPE_EXCEEDED` 기록

## 6. Escalation Rule

### 6.1 트리거별 Escalation 경로

| 트리거 | Risk 등급 | Escalation 경로 | 즉각 조치 |
|--------|----------|----------------|----------|
| **Risk CRITICAL 감지** | Critical | 즉시 중단 → Orch → Business Owner → Executive Sponsor | 전체 Agent 중단 + 격리 |
| **Risk HIGH + 미승인 실행** | High | 실행 차단 → Human 승인 게이트 → AI Architect | 승인 게이트 재진입 |
| **KPI Threshold 초과** | High | Monitor-A → Orch → AI Architect → 72h 내 RCA | 알림 발송 + 로그 기록 |
| **Drift 감지** | Medium | Monitor-A → Orch → Evol-A 제안 → Governance 승인 | Evolution Loop 시작 |
| **Scope 초과 시도** | High | 즉시 차단 → Orch → AI Architect → 예외 승인 절차 | 실행 차단 + 로그 |
| **Audit Log 실패** | High | 배포 차단 → Back-A 재작업 → AI Architect 재심사 | 배포 파이프라인 중단 |
| **Context 무단 수정 시도** | Critical | 즉시 차단 → Orch → Governance Board 보고 → Context 복원 | Context 복원 + 감사 |
| **Gate SLA 타임아웃** | Medium | 자동 리마인더 → +24h → PM 에스컬레이션 | 리마인더 발송 |
| **Tool 호출 3회 연속 실패** | High | Orch 에스컬레이션 → AI Architect → 수동 점검 | Tool 일시 중단 |
| **Override 발동** | Critical | 전체 Agent 중단 (3초) → Human 조치 → 24h 내 Governance 보고 | 즉시 전파 |

### 6.2 Escalation SLA

| 등급 | 응답 SLA | 해결 SLA | 미해결 시 조치 |
|------|---------|---------|--------------|
| **Critical** | 1시간 | 4시간 | Executive Sponsor 즉시 보고 + 시스템 격리 |
| **High** | 4시간 | 24시간 | Governance Board 보고 + 임시 운영 중단 |
| **Medium** | 업무일 24시간 | 업무일 72시간 | AI Architect 검토 + 개선 계획 수립 |
| **Low** | 업무일 48시간 | 업무일 7일 | 정기 리뷰 회의에서 처리 |

## 7. Agent Interaction Spec

### 7.1 메시지 구조 표준

모든 Agent 간 메시지는 다음 표준 구조를 따른다. Phase 3에서 `AgentMessage` 타입으로 구현한다.

| 필드 | 타입 | 설명 | 필수 여부 |
|------|------|------|----------|
| `message_id` | UUID | 메시지 고유 식별자 | ✅ 필수 |
| `sender_agent` | string | 발신 Agent 코드 (Orch / Back-A 등) | ✅ 필수 |
| `receiver_agent` | string | 수신 Agent 코드 또는 `'human'` | ✅ 필수 |
| `message_type` | enum | `REQUEST / RESPONSE / ESCALATION / NOTIFICATION` | ✅ 필수 |
| `payload` | JSON | 실제 데이터 (요청 내용 또는 결과) | ✅ 필수 |
| `context_version` | string | 참조한 Global Context 버전 (e.g. `v1.1`) | ✅ 필수 |
| `risk_level` | enum | `LOW / MEDIUM / HIGH / CRITICAL` | ✅ 필수 |
| `requires_approval` | boolean | Human 승인 필요 여부 | ✅ 필수 |
| `created_at` | datetime | 메시지 생성 시각 (UTC) | ✅ 필수 |
| `trace_id` | UUID | 전체 Workflow 추적 ID (동일 요청 체인 공유) | ✅ 필수 |
| `parent_message_id` | UUID | 부모 메시지 ID (응답 체인 추적용) | 선택 |
| `ttl_seconds` | integer | 메시지 유효 시간(초). 초과 시 타임아웃 처리 | 선택 |

### 7.2 Agent 간 통신 규칙

| 규칙 | 내용 | 위반 처리 |
|------|------|----------|
| **Orchestrator 경유 원칙** | Capability Agent 간 직접 통신 금지. 반드시 Orch 경유 | 메시지 차단 + Orch 알림 |
| **trace_id 공유 필수** | 동일 요청 체인의 모든 메시지는 같은 `trace_id` 사용 | 추적 불가 메시지 거부 |
| **context_version 일치** | 메시지의 `context_version`이 현재 Global Context와 일치해야 함 | 버전 불일치 시 재로드 후 재시도 |
| **risk_level 명시 의무** | 모든 메시지에 `risk_level` 필수 포함 | 메시지 거부 + 로그 |
| **응답 SLA 준수** | 메시지 수신 후 SLA 내 응답 없으면 자동 타임아웃 처리 | Escalation 자동 트리거 |
| **Human 메시지 최우선** | Human에서 오는 메시지는 Agent 메시지보다 항상 우선 처리 | 해당 없음 (설계 원칙) |

## 8. 리스크 통제 적용

> 기준 문서: `0_4 리스크 관리 표준` + `governance_rules_context.json`  
> ※ Cursor AI 코드 생성 시 `@governance_rules_context.json` 참조 필수 (리스크 임계값·Zero-Trust 규칙 포함)

### 8.1 Workflow 내 필수 통제 항목

| 통제 항목 | 통제 유형 | 적용 위치 | 검증 방법 |
|----------|----------|----------|----------|
| **Authority Level 검증** | Preventive | 모든 Agent 실행 전 | `governance.json` 로드 후 level 확인 |
| **Scope 검증** | Preventive | Tool 호출 전 | `allowed_tools` 목록 대조 |
| **Hallucination 테스트** | Detective | LLM 결과 수신 후 | Test-A 자동 검증 시나리오 |
| **Bias 테스트** | Detective | Persona 분석 / 추천 생성 후 | Test-A 편향성 체크 |
| **Drift Detection** | Detective | Phase 5 배포 후 (Monitor-A) | KPI Threshold 비교 |
| **Prompt Injection 방어** | Preventive | 외부 입력값 수신 시 | 입력값 검증 미들웨어 |
| **Audit Log 활성화** | Detective | 모든 Agent 실행 결과 | `audit_logs` INSERT 확인 |

### 8.2 리스크 분류 체계 적용

| 리스크 유형 | 해당 Agent | 주요 리스크 항목 | 완화 방법 |
|-----------|-----------|----------------|----------|
| **기술적 리스크** | 모든 LLM 호출 Agent | Hallucination / Bias / Drift / Prompt Injection | Test-A 검증 + Monitor-A 탐지 |
| **운영 리스크** | Orch / Back-A | Scope 위반 / 권한 오남용 / 로그 미기록 | Scope 검증 로직 + Audit Log 강제 |
| **법적/규제 리스크** | Evol-A / Orch | 개인정보 침해 / 차별 / 규제 미준수 | Compliance Officer 검토 + RBAC |
