## 1. AI 리스크 관리 원칙

### 1.1 AI 리스크 관리 철학

AI 시스템의 리스크는 **제거할 수 없다. 관리할 수 있을 뿐이다.**

이 원칙의 기반:
- 모든 AI 리스크는 탐지 가능한 신호를 가진다
- 리스크는 수준별로 차등 통제한다 (Risk-Based Control)
- 인간 개입(Human-in-the-Loop)은 리스크 통제의 최후 보루이다
- 리스크 통제 기록 자체가 감사 가능성(Auditability)의 증거이다

---

### 1.2 AI 리스크 유형별 상세 정의

#### 리스크 1: Bias (편향)

| 항목 | 내용 |
|------|------|
| **정의** | 학습 데이터 또는 모델 구조로 인해 특정 집단·상황에 불공정한 결과가 반복적으로 발생하는 현상 |
| **발생 원인** | 편향된 학습 데이터, 대표성 부족 데이터, 모델 구조 편향 |
| **탐지 신호** | 특정 집단의 승인율/거부율 편차, 응답 품질 집단 간 차이, 동일 입력 조건 다른 결과 |
| **탐지 임계값** | 집단 간 결과 편차 ±10% 초과 시 → Bias 검토 트리거 |
| **완화 전략** | ① 다양성 있는 학습 데이터 구성, ② 정기적 Bias 감사(분기 1회), ③ fairness_check() 검증 로직 필수 |
| **허용 기준** | Bias 지표 ±5% 이내 → 허용 / ±5~10% → 모니터링 강화 / ±10% 초과 → 즉시 중단·재검토 |
| **책임 소재** | AI Architect (탐지 구조 설계) + Compliance Officer (법적 영향 검토) |

---

#### 리스크 2: Hallucination (환각)

| 항목 | 내용 |
|------|------|
| **정의** | LLM 기반 모델이 사실과 다른 정보를 자신 있게 생성하는 현상 |
| **발생 원인** | LLM의 확률적 생성 특성, 학습 데이터 한계, 검증 부재 |
| **탐지 신호** | 검증 불가능한 출처 인용, 모순된 응답 반복, 실제 데이터와 불일치 |
| **탐지 임계값** | 사실 검증 실패율 5% 초과 → Hallucination 경보 |
| **완화 전략** | ① validate_response() 필수 적용, ② RAG(검색 보강 생성) 구조 우선 사용, ③ 중요 의사결정에 LLM 단독 사용 금지, ④ Human 검토 게이트 필수 |
| **허용 기준** | 검증 실패율 1% 미만 → 허용 / 1~5% → 경보 + 추가 검증 / 5% 초과 → 배포 금지 |
| **책임 소재** | Back-A (validate_response 구현) + Human Reviewer (최종 확인) |

---

#### 리스크 3: Explainability (설명 가능성 부재)

| 항목 | 내용 |
|------|------|
| **정의** | AI 의사결정 과정이 불투명하여 왜 그런 결과가 나왔는지 설명할 수 없는 상태 |
| **발생 원인** | 복잡한 딥러닝 모델, 설명 구조 미설계, 블랙박스 API 의존 |
| **탐지 신호** | 규제기관 설명 요청 불응 상황, 감사 시 근거 제시 불가, 이해관계자 이의 대응 불가 |
| **탐지 임계값** | 의사결정 설명 성공률 < 90% → 즉시 검토 |
| **완화 전략** | ① 모든 AI 의사결정에 decision_basis 필드 기록 필수, ② 간단한 모델 우선 사용(해석 가능성 중시), ③ 설명 불가 의사결정은 Human 직접 결정으로 대체, ④ SHAP/LIME 등 설명 도구 연동 |
| **허용 기준** | 모든 사용자 영향 의사결정 100% 설명 가능해야 함 — 예외 없음 |
| **책임 소재** | AI Architect (설명 구조 설계) + AI Governance Board (기준 승인) |

---

#### 리스크 4: Drift (성능 저하)

| 항목 | 내용 |
|------|------|
| **정의** | 시간 경과·환경 변화에 따라 모델의 성능이 점진적으로 저하되는 현상 |
| **발생 원인** | 데이터 분포 변화(Data Drift), 개념 변화(Concept Drift), 운영 환경 변화 |
| **탐지 신호** | KPI 연속 하락, 예측 정확도 감소, 이상 응답 빈도 증가, 입력 데이터 분포 변화 |
| **탐지 임계값** | KPI 목표 대비 10% 이상 하락 → Drift 경보 / 20% 이상 → 즉시 재평가 |
| **완화 전략** | ① Monitor-A를 통한 실시간 KPI 모니터링 필수, ② 정기 모델 재검증(월 1회), ③ Drift 탐지 시 자동 알림 → Evolution Agent 제안 → Governance Board 검토, ④ 모델 버전 관리 및 롤백 경로 확보 |
| **허용 기준** | KPI 목표 대비 ±10% 이내 → 허용 / 10~20% 하락 → 개선 계획 수립 / 20% 초과 → 모델 교체 |
| **책임 소재** | Monitor-A (자동 탐지) + AI Architect (재평가) + Evolution Agent (개선 제안) |

---

#### 리스크 5: Ethics / Accountability (윤리·책임)

| 항목 | 내용 |
|------|------|
| **정의** | 자동화된 AI 의사결정으로 인한 윤리적 문제 또는 책임 귀속 불명확 상태 |
| **발생 원인** | 과도한 자동화, 책임 구조 불명확, 윤리 검토 절차 부재 |
| **탐지 신호** | 이해관계자 민원 발생, 미디어 부정적 보도, 규제기관 조사, 내부 윤리 위원회 문제 제기 |
| **탐지 임계값** | 윤리 민원 1건 발생 → 즉시 조사 / 반복 발생 → 시스템 구조 재검토 |
| **완화 전략** | ① 모든 최종 결정은 Human에 귀속 (Agent는 Accountable 역할 불가), ② 윤리 영향 평가를 배포 전 필수 수행, ③ Human Override는 언제든지 가능 (이유 불필요), ④ 윤리 검토 Compliance Officer 참여 필수 |
| **허용 기준** | Human 책임 귀속 구조 없이는 배포 불가 — 예외 없음 |
| **책임 소재** | Compliance Officer (윤리 검토) + Governance Board (배포 승인) + Business Owner (최종 책임) |

### 1.3 AI 리스크 종합 관리 매트릭스

| 리스크 유형 | 심각도 | 탐지 주기 | 관리 주체 | 통제 수준 |
|-----------|--------|---------|---------|---------|
| Bias | HIGH | 분기 1회 + 이상 징후 시 즉시 | AI Architect + Compliance Officer | 강화 통제 |
| Hallucination | HIGH | 실시간 (validate_response) | Back-A + Human Reviewer | 강화 통제 |
| Explainability | CRITICAL | 배포 전 100% 검증 | AI Architect + Governance Board | 최고 수준 통제 |
| Drift | MEDIUM→HIGH | 실시간 모니터링 (Monitor-A) | Monitor-A + Evolution Agent | 표준→강화 통제 |
| Ethics/Accountability | CRITICAL | 배포 전 검토 + 운영 중 지속 | Compliance Officer + Business Owner | 최고 수준 통제 |

### 1.4 AI 리스크 평가 절차

리스크 이벤트 발생 시 아래 절차를 따른다.

```
[1단계] 탐지
    Monitor-A 또는 Human이 리스크 신호 감지
    → 즉시 override_logs 또는 audit_logs에 기록

[2단계] 분류
    AI Architect가 리스크 유형 및 심각도 분류
    → HIGH/CRITICAL: 즉시 실행 중단
    → MEDIUM 이하: 경보 발송 + 모니터링 강화

[3단계] 완화 조치
    해당 리스크 완화 전략 즉시 적용
    → Orchestrator에 완화 조치 승인 요청

[4단계] 재평가
    완화 조치 48시간 후 리스크 재평가
    → 허용 기준 내 진입 시 → 운영 재개
    → 미진입 시 → Governance Board 보고 + 구조 개선

[5단계] 기록
    전체 과정 audit_logs 기록 필수
    → trajectory_logs에 실패 패턴 태깅 (failure_tagged = true)
    → Evolution Agent 입력값으로 활용
```

## 2. Zero-Trust 보안 설계 원칙

### 2.1 Zero-Trust 핵심 선언

> **"Never Trust, Always Verify"**
> AI 시스템에서 신뢰는 위치(네트워크)가 아닌 검증(Identity + Context)으로 결정된다.

기존 보안 모델의 한계:
- 내부 네트워크는 안전하다는 가정 → AI 시스템에서 무효
- 한번 인증하면 계속 신뢰 → AI Agent 실행 중 권한 오용 가능
- 경계 기반 방어 → AI 클라우드·API 환경에서 경계 불명확

Zero-Trust가 AI 시스템에 필수인 이유:
- AI Agent는 API를 통해 다양한 외부 서비스에 접근
- LLM은 Prompt Injection으로 권한 우회 시도 가능
- 자동화된 실행은 사람이 실시간 감시 불가

---

### 2.2 Zero-Trust 5대 설계 원칙

#### 원칙 ZT-1: Identity First (신원 우선 검증)

```
모든 요청은 발신자 신원을 먼저 검증한다.
네트워크 위치(내부/외부)는 신뢰의 근거가 되지 않는다.

설계 적용:
- 모든 API 요청: JWT 토큰 검증 필수 (유효 기간 15분)
- Agent 간 통신: 서비스 토큰 검증 필수
- 내부 서비스라도 인증 없이 신뢰하지 않는다
- 모든 인증 실패: audit_logs 즉시 기록
```

#### 원칙 ZT-2: Least Privilege (최소 권한)

```
각 주체(Agent, User, Service)는 현재 작업에 필요한 최소 권한만 가진다.
권한은 요청 시점에 동적으로 검증한다.

설계 적용:
- Agent별 allowed_scope 명시 (governance.json 참조)
- 권한 상승은 Governance Board 승인 후에만 가능
- 작업 완료 후 임시 권한은 즉시 만료
- Scope 초과 접근 시도: 즉시 차단 + override_logs 기록
```

#### 원칙 ZT-3: Micro-Segmentation (마이크로 분리)

```
시스템을 최소 단위로 분리하여 침해 시 전파 범위를 제한한다.
하나의 Agent가 침해되더라도 전체 시스템에 영향이 없어야 한다.

설계 적용:
- Agent는 자신의 DB 테이블에만 접근 (타 Agent DB 접근 금지)
- 서비스 간 통신: API Gateway를 반드시 경유
- 데이터 분류(Public/Internal/Sensitive/Restricted)에 따른 접근 경계 설정
- 에러 격리: 한 Agent 장애가 전체 시스템 장애로 전파되지 않는 구조
```

#### 원칙 ZT-4: Continuous Verification (지속적 검증)

```
인증은 일회성이 아니다. 실행 중에도 지속적으로 권한을 재검증한다.
상황 변화 시 즉시 권한을 재평가한다.

설계 적용:
- 장기 실행 Agent: 15분마다 Token 재검증
- Risk Level 상승 감지 시: 즉시 권한 재평가 (on_risk 체크)
- 비정상 패턴 감지 시: 자동 권한 정지 + Human 개입 요청
- 모든 권한 재검증 결과: audit_logs 기록
```

#### 원칙 ZT-5: Assume Breach (침해 가정)

```
시스템이 이미 침해되었다고 가정하고 설계한다.
이 가정 하에 피해 최소화·탐지·복구 구조를 우선 설계한다.

설계 적용:
- 모든 실행: 사후 감사 가능하도록 audit_logs 기록
- LLM 입력: Prompt Injection 가능성 전제 → sanitize_input() 필수
- 외부 API 응답: 신뢰하지 않고 validate_response() 검증
- 시스템 상태 스냅샷: 비정상 감지 시 즉시 저장 (복구 기반)
- Human Override: 언제든지 즉시 가능 (3초 내 전체 중단)
```

### 2.3 Zero-Trust 신뢰 경계 정의

AI 시스템의 신뢰 경계는 다음과 같이 정의한다.

```
┌─────────────────────────────────────────────────────┐
│                  External Zone (신뢰 없음)            │
│   사용자 입력, 외부 API, 외부 LLM, 외부 데이터 소스   │
│   → 모든 입력: sanitize_input() 검증 필수             │
│   → 모든 응답: validate_response() 검증 필수          │
└──────────────────┬──────────────────────────────────┘
                   │ API Gateway (신뢰 경계 통문)
                   │ JWT 검증 + Rate Limiting + Allowlist
┌──────────────────▼──────────────────────────────────┐
│               Internal Service Zone (조건부 신뢰)    │
│   Orchestrator, Agent 서비스, Internal API           │
│   → 서비스 토큰 검증 필수 (내부도 신뢰 없음)          │
│   → RBAC 기반 최소 권한 적용                         │
│   → 모든 서비스 간 통신: TLS 필수                    │
└──────────────────┬──────────────────────────────────┘
                   │ DB Access Layer
                   │ Tenant Filter + Soft Delete Filter
┌──────────────────▼──────────────────────────────────┐
│                  Data Zone (최고 보호)               │
│   PostgreSQL, Redis, audit_logs, 민감 데이터         │
│   → Sensitive 이상: 암호화 저장 필수                 │
│   → audit_logs: INSERT ONLY (DELETE/UPDATE 절대 금지) │
│   → Restricted: 암호화 + 익명화 + 접근 로그 3중 보호  │
└─────────────────────────────────────────────────────┘
```

### 2.4 LLM 환경의 Zero-Trust 특수 원칙

LLM은 일반 소프트웨어와 달리 입력에 의해 동작이 변경될 수 있다.

| 위협 | Zero-Trust 대응 설계 |
|-----|---------------------|
| Prompt Injection | 사용자 입력은 시스템 프롬프트와 완전 분리 + sanitize_input() |
| 권한 우회 시도 | LLM 출력은 코드 실행 전 validate_response() 검증 |
| 시스템 프롬프트 노출 | 환경변수 또는 context JSON 로드 (하드코딩 절대 금지) |
| 민감 정보 유출 | LLM에 전달하는 데이터는 Internal 이하만 허용 |
| 자율 실행 시도 | LLM이 제안한 모든 실행 → Human 승인 게이트 통과 필수 |

## 3. Risk-Based Approval 프로세스

### 3.1 Risk-Based Approval 설계 철학

승인은 **리스크에 비례**해야 한다.
- 낮은 리스크 → 가벼운 승인 (업무 속도 확보)
- 높은 리스크 → 강화된 승인 (안전성 확보)
- 리스크 없는 승인 요구는 관료주의이다
- 승인 없는 리스크 실행은 거버넌스 붕괴이다

---

### 3.2 승인 필요 여부 결정 매트릭스

행동 유형(Action Type)과 리스크 수준(Risk Level)의 교차점이 필요 승인 수준을 결정한다.

| 행동 유형 \ Risk Level | LOW | MEDIUM | HIGH | CRITICAL |
|----------------------|-----|--------|------|----------|
| 데이터 조회 (SELECT) | 불필요 | 불필요 | 로그 기록 | HITL P2 즉시 |
| 분석/추천 생성 | 불필요 | 로그 기록 | HITL P1 | HITL P1 + Board |
| 데이터 변경 (INSERT/UPDATE) | 로그 기록 | HITL P1 | HITL P1 + 2중 승인 | 즉시 중단 |
| AI 모델 실행 | 로그 기록 | HITL P1 | HITL P1 + Risk 체크 | 즉시 중단 + Board |
| Context 수정 | HITL P1 | HITL P1 + Board | 즉시 중단 + Board | 배포 금지 |
| 외부 API 호출 | Allowlist 검사 | HITL P1 | HITL P1 + 감사 | 즉시 중단 |
| 배포 (Deploy) | HITL P3 | HITL P3 + Board | HITL P3 + Board + 2중 | 배포 금지 |

**범례:**
- `불필요`: 승인 없이 실행 가능 (단, 로그 기록 필수)
- `로그 기록`: audit_logs INSERT 후 실행
- `HITL P1`: Pre-approval — 실행 전 Human 승인 필수 (SLA 24h)
- `HITL P2`: On-risk — 실행 중 이상 감지 시 즉시 중단 + 에스컬레이션 (SLA 1h)
- `HITL P3`: Post-review — 산출물 완료 후 Human 검토 필수 (SLA 48h)
- `Board`: Governance Board 추가 승인 필수
- `즉시 중단`: Agent 자동 중단 → Override 절차 적용

---

### 3.3 승인 요청 생애주기 (Lifecycle)

#### P1 Pre-approval 생애주기

```
[CREATED]
    Agent가 실행 전 승인 요청 생성 → approval_requests INSERT
    status = PENDING
       │
       ├── 24h 이내 승인 → [APPROVED] → Agent 실행 재개
       │
       ├── 24h 이내 거부 → [REJECTED] → 실행 취소 + 사유 기록
       │
       └── 24h 초과 미응답 → [ESCALATED]
               → 상위 승인자에게 자동 이관
               → +24h 초과 → PM 직접 알림
```

#### P2 On-risk 생애주기

```
[TRIGGERED]
    Monitor-A 또는 Agent가 Risk 감지 → override_logs INSERT
    status = TRIGGERED
       │
       ├── 1h 이내 Human 대응 → [RESOLVED]
       │       → 원인 분석 + 대응 조치 기록
       │       → 재개 또는 롤백 결정
       │
       └── 1h 초과 미응답 → [ESCALATED]
               → PM 즉시 호출 + Business Owner 알림
               → Governance Board 24h 내 보고 의무
```

#### P3 Post-review 생애주기

```
[SUBMITTED]
    Agent 산출물 완료 → review_requests INSERT
    status = PENDING
       │
       ├── 48h 이내 확정 → [CONFIRMED] → 산출물 배포/적용 가능
       │
       ├── 48h 이내 수정 요청 → [REVISED] → Agent 수정 후 재제출
       │
       ├── 48h 이내 거부 → [REJECTED] → 산출물 폐기 + 재설계
       │
       └── 48h 초과 미응답
               → 리마인더 발송
               → +24h 초과 → AI Architect → PM 에스컬레이션
```

---

### 3.4 승인자 및 에스컬레이션 매트릭스

| 승인 단계 | 1차 승인자 | 에스컬레이션 (SLA 초과 시) | 최종 에스컬레이션 |
|---------|-----------|----------------------|---------------|
| L0~L1 일반 | Business Owner | AI Architect | PM |
| L1 HIGH Risk | Business Owner | AI Governance Board | Executive Sponsor |
| L2 조건부 자동 | AI Governance Board | Executive Sponsor | 불가 (즉시 중단) |
| L3 고도 자율 | Executive + Governance Board 공동 | 불가 (즉시 중단) | - |
| Context 수정 | Governance Board | Executive Sponsor | 불가 (즉시 중단) |
| CRITICAL Risk | 즉시 중단 (자동) | Business Owner 즉시 알림 | Governance Board 24h 내 |

### 3.5 예외 승인 프로세스 (Exception Approval)

정규 승인 절차를 따를 수 없는 긴급 상황에 대한 예외 처리 절차이다.

**예외 승인이 허용되는 조건:**
- 보안 침해 대응을 위한 긴급 조치 필요 시
- 규제기관 즉각 대응 요구 시
- 시스템 장애로 인한 긴급 복구 필요 시
- 기타 Governance Board가 승인한 특수 상황

**예외 승인 절차:**

```
Step 1: 예외 요청 제출
        - 요청자: AI Architect 또는 Business Owner
        - 내용: 예외 사유, 영향 범위, 완화 조치, 유효 기간(최대 90일)
        - 즉시 audit_logs에 EXCEPTION_REQUEST 기록

Step 2: Governance Board 긴급 승인
        - SLA: 4시간 (긴급 상황)
        - 승인 조건: 완화 조치 명시 필수
        - 미응답 시: Business Owner 단독 임시 승인 가능 (단, 24h 내 Board 추인 필수)

Step 3: 예외 실행
        - audit_logs에 EXCEPTION_APPROVED + 승인자 기록
        - 모든 예외 실행은 상시 Human 모니터링 하에 진행

Step 4: 사후 보고
        - 예외 종료 후 5영업일 이내 전체 경위 보고서 제출
        - trajectory_logs에 failure_tagged = true 기록
        - Evolution Agent 입력값으로 활용

Step 5: 구조 개선 여부 결정
        - 동일 예외 3회 이상 반복 시 → 구조적 개선 필수 (Scope 변경 또는 프로세스 개선)
```

**예외 승인 금지 사항:**
- CRITICAL Risk 상황에서의 예외 실행 (즉시 중단이 원칙)
- Audit Log 없는 예외 실행
- 유효 기간 명시 없는 무기한 예외
- 동일 예외의 무한 반복 (3회 초과 → 구조 개선 의무)

### 3.6 Approval Audit Trail (승인 감사 추적)

모든 승인 이력은 아래 기준으로 기록되어야 한다.

| 항목 | 필수 기록 내용 |
|------|-------------|
| 승인 요청 | agent_id, action_type, risk_level, payload_hash, requested_by, requested_at |
| 승인 결정 | approver, decision(APPROVED/REJECTED), decision_basis, decided_at |
| 에스컬레이션 | escalated_to, escalation_reason, escalated_at |
| 예외 승인 | exception_reason, validity_period, mitigation_measures, board_approval |
| Override | triggered_by, trigger_reason, affected_agents, snapshot_ref, resolved_at |

**기록 원칙:**
- 모든 승인 기록: INSERT ONLY (수정·삭제 절대 금지)
- 보관 기간: 최소 5년
- 감사 요청 시 72시간 이내 제공 가능한 구조

## 4. Governance 규칙 위반 시 처리

### 4.1 위반 유형별 즉각 대응

| 위반 유형 | 즉각 조치 | 48h 내 후속 조치 |
|---------|---------|---------------|
| AI 리스크 임계값 초과 | Agent 실행 중단 + 경보 | 원인 분석 + 완화 조치 + Governance 보고 |
| Zero-Trust 검증 실패 | 요청 차단 + 보안 로그 기록 | 침해 경로 분석 + 보안 강화 |
| 승인 없는 HIGH Risk 실행 | 롤백 + 즉시 중단 | 승인 게이트 강화 + 책임자 확인 |
| Audit Log 누락 | 배포 차단 | 로그 자동 검증 체계 추가 |
| Prompt Injection 감지 | 요청 차단 + 격리 | 입력 검증 로직 강화 |
| 예외 승인 무한 반복 (3회+) | 구조 개선 의무화 | Scope 변경 또는 프로세스 개선 |

### 4.2 반복 위반 처리 원칙

동일한 유형의 위반이 3회 이상 반복되는 경우:
1. 단순 대증적 조치로는 종결 불가
2. Governance Board가 구조적 원인 분석 수행
3. 프레임워크 설계 변경 또는 프로세스 재정의 필수
4. Evolution Agent가 개선안 제안 → Board 승인 후 적용
