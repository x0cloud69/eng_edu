## 2. Scope 경계 원칙

### 2.1 핵심 원칙

이 프로젝트의 Scope는 아래 3가지 원칙으로 결정된다.

**원칙 1 — Framework Level Only**
> 이 프로젝트는 **표준 구조 및 거버넌스 설계**를 목표로 한다.
> 실제 시스템 구축(Implementation)은 Scope 외부이다.
> 만들어야 할 것: "어떻게 만들어야 하는가의 표준"
> 만들지 않을 것: "특정 목적을 위한 실제 서비스"

**원칙 2 — Reusability First**
> 모든 In-Scope 항목은 **다수의 프로젝트에 재사용 가능**해야 한다.
> 단 하나의 프로젝트 또는 기관에만 적용 가능한 것은 Out-of-Scope이다.

**원칙 3 — Governance Boundary**
> AI 통제·감사·승인 체계와 직접 연관된 것은 In-Scope이다.
> AI 통제와 무관한 비즈니스 기능·UI·인프라는 Out-of-Scope이다.

### 2.2 Scope 판단 3문항

어떤 항목이 Scope 내인지 판단할 때 아래 3가지를 묻는다.

```
Q1. 이것이 표준 구조 / 거버넌스 설계인가? (Yes → 다음 질문)
Q2. 2개 이상의 프로젝트에 재사용 가능한가?  (Yes → 다음 질문)
Q3. AI 통제·감사·승인과 직접 연관되는가?  (Yes → In-Scope 후보)

세 질문 중 하나라도 No → Out-of-Scope 검토 필요
```

| **향후 처리** | 법무팀이 규제 해석 후 해당 내용을 프레임워크 정책에 반영하는 방식으로 협력 |

## 5. Scope 판단 기준 (Decision Tree)

새로운 작업 항목이 발생했을 때 In/Out 판단하는 의사결정 흐름이다.

```
새 작업 항목 발생
       │
       ▼
[Q1] 이것이 표준 구조·거버넌스·공통 설계인가?
       │
       ├─ NO ──→ Out-of-Scope (실제 구현, 특정 서비스)
       │
       └─ YES
              │
              ▼
       [Q2] 2개 이상 프로젝트에 재사용 가능한가?
              │
              ├─ NO ──→ Out-of-Scope (특정 기관 전용)
              │
              └─ YES
                     │
                     ▼
              [Q3] AI 통제·감사·승인 체계와 직접 연관되는가?
                     │
                     ├─ NO ──→ 보류 (Governance Board 판단 요청)
                     │
                     └─ YES
                            │
                            ▼
                     [Q4] IS-01~IS-09 중 기존 항목의 일부인가?
                            │
                            ├─ YES ──→ In-Scope (기존 항목 내 포함)
                            │
                            └─ NO ──→ Scope 변경 요청 절차 (Section 7)
```

## 6. Phase별 Scope 적용 시점

이 문서의 Scope 정의는 아래 Phase 계획에 따라 구체화된다.

| Phase | 적용 범위 | In-Scope 항목 |
|-------|---------|--------------|
| Phase 1 (Vision & Role) | 설계 및 정의 | IS-01~IS-09 전체 정의 완료 |
| Phase 2 (Workflow) | 워크플로우 설계 | IS-03, IS-06 워크플로우 상세화 |
| Phase 3 (Capability) | 구현 기준 정의 | IS-01, IS-02, IS-05, IS-07 코드 기준 확정 |
| Phase 4 (Simulation) | 검증 | IS-09 파일럿 검증 기준 적용 |
| Phase 5 (Deploy) | 배포 | IS-08 KPI 측정 시작 |
| Phase 6 (Evolution) | 개선 | 전체 Scope 재검토 및 업데이트 |

> ※ Out-of-Scope(OS-01~OS-09)는 해당 Phase에서도 프레임워크 범위 외로 유지된다.
> 각 도메인 프로젝트 또는 실행 단계에서 이 프레임워크를 기반으로 구현한다.

## 7. Scope 변경 통제 절차

Scope 변경은 아래 절차를 통해서만 허용된다.

### 7.1 변경 요청 조건

다음 중 하나에 해당하는 경우 Scope 변경 요청이 가능하다.

1. 새로운 규제 요건이 추가되어 기존 Scope로 충족 불가능할 때
2. 이해관계자 목표(1.1.2)가 변경되어 Scope 재정의가 필요할 때
3. Phase 4 파일럿 검증 결과 IS 항목 조정이 필요할 때
4. Drift 또는 KPI 미달로 구조적 Scope 확장이 필요할 때

### 7.2 변경 절차

```
Step 1: 변경 요청서 작성
        - 변경 항목, 변경 이유, 영향 범위, 대안 분석 포함

Step 2: AI Architect 검토 (3영업일 이내)
        - 기술적 타당성 및 프레임워크 원칙 부합 여부 확인

Step 3: Business Owner 승인
        - 비용·일정 영향 판단

Step 4: Governance Board 최종 승인
        - 거버넌스 원칙 위반 여부 최종 확인

Step 5: 문서 업데이트
        - 이 문서 버전 증가 + change_log 기록
        - scope_context.json 업데이트
        - 07_scope_governance.mdc 업데이트
        - project_context.json scope 섹션 업데이트

Step 6: 영향받는 팀에 변경 공지
```

### 7.3 Scope Creep 방지 원칙

> Scope Creep이란 명확한 승인 없이 Scope가 점진적으로 확장되는 현상이다.

- **의심 신호:** "이것도 만들어 두면 좋을 것 같은데" 요청
- **대응 원칙:** Section 5 Decision Tree 즉시 적용 → Out-of-Scope 확인 시 즉시 거부
- **기록:** 모든 Out-of-Scope 요청은 Backlog에 기록 (향후 Phase에서 재검토)

## 8. Scope 위반 시 처리 기준

Out-of-Scope 항목이 구현된 것이 발견된 경우 아래 기준으로 처리한다.

| 위반 유형 | 처리 방법 | 담당자 |
|---------|---------|-------|
| Out-of-Scope 코드가 PR에 포함된 경우 | PR 반려, 코드 제거 요청 | AI Architect |
| 이미 배포된 Out-of-Scope 기능 발견 | 롤백 또는 분리 프로젝트로 이전 | Governance Board |
| Scope 변경 없이 IS 항목 확장된 경우 | 설계 문서 수정 + Governance 소급 승인 | Business Owner |
| 반복적 Scope 위반 발생 시 | Scope 변경 절차 검토 또는 WBS 재정의 | AI Architect + Business Owner |
